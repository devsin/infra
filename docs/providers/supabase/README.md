# Supabase Platform Management

## Overview

Each brand has its **own Supabase project** (or set of projects per environment). The parent company is an admin/owner in each, but they are separate tenants.

[Supabase](https://supabase.com) provides a managed backend stack: PostgreSQL database, authentication, storage, edge functions, and realtime subscriptions. This Terraform stack manages:
- Supabase projects (one per brand or per brand-env)
- Database configuration
- Auth providers and settings
- Storage buckets
- Edge function deployment settings
- Custom domains

## Architecture

```
<parent-org> (admin)
│
├── Supabase Project: <brand-a>-prod  ← Brand A production
│   ├── Database, Auth, Storage
│   └── managed via envs/<brand-a>.tfvars
│
├── Supabase Project: <brand-a>-dev   ← Brand A development
│   └── managed via envs/<brand-a>.tfvars (with env toggle)
│
├── Supabase Project: <brand-b>-prod  ← Brand B production
│   └── managed via envs/<brand-b>.tfvars
│
└── ...
```

> **Key point:** Each Supabase project is an independent tenant with its own
> database, auth, and API keys. The parent org manages them all via Terraform
> using a management API token.

## Terraform Provider

Supabase has a first-party Terraform provider:

```hcl
# versions.tf
terraform {
  required_version = ">= 1.10.0"

  required_providers {
    supabase = {
      source  = "supabase/supabase"
      version = "~> 1.0"
    }
  }
}
```

> Check [registry.terraform.io/providers/supabase/supabase](https://registry.terraform.io/providers/supabase/supabase/latest)
> for the latest version.

## Authentication

Supabase Terraform provider uses an **access token** from the Supabase dashboard.

```hcl
# providers.tf
provider "supabase" {
  access_token = var.supabase_access_token  # or SUPABASE_ACCESS_TOKEN env var
}
```

> **Security:** Never commit tokens. Use environment variables or a secrets manager.

## Key Variables

```hcl
# variables.tf
variable "brand_name" {
  description = "Brand name (must match brands.yaml)"
  type        = string
}

variable "supabase_access_token" {
  description = "Supabase management API access token"
  type        = string
  sensitive   = true
}

variable "organization_id" {
  description = "Supabase organization ID"
  type        = string
}

variable "projects" {
  description = "Map of Supabase projects to create"
  type = map(object({
    region          = optional(string, "eu-west-1")
    database_password = optional(string, null)  # generated if null
    plan            = optional(string, "free")   # free, pro, team
    environments    = optional(list(string), ["production"])
  }))
  default = {}
}
```

### Example Brand tfvars


## What Gets Managed

| Resource | Managed by Terraform? | Notes |
|----------|----------------------|-------|
| Organization | ❌ No | Created manually in Supabase dashboard |
| Projects | ✅ Yes | Created and configured |
| Database schema | ❌ No | Managed via migrations (supabase CLI) |
| Auth providers | ⚠️ Partial | Provider supported, but complex configs |
| Storage buckets | ✅ Yes | If provider supports it |
| Edge functions | ❌ No | Deployed via supabase CLI or CI/CD |
| Custom domains | ✅ Yes | Attached to projects |
| API keys | ✅ Output | Generated by project creation |

## Deployment

```bash
cd supabase/stacks/brand
terraform init -backend-config="prefix=providers/supabase/brand/<brand>"
terraform plan -var-file=envs/<brand>.tfvars
terraform apply -var-file=envs/<brand>.tfvars
```

## Supabase vs AWS/GCP Backend Services

| Feature | Supabase | AWS | GCP |
|---------|----------|-----|-----|
| Database | Managed Postgres | RDS / Aurora | Cloud SQL |
| Auth | Built-in (GoTrue) | Cognito | Firebase Auth / Identity Platform |
| Storage | Built-in (S3-compatible) | S3 | GCS |
| Edge Functions | Deno-based | Lambda@Edge | Cloud Functions |
| Realtime | Built-in | AppSync / IoT | Firestore |

### When to Use Supabase

- Rapid prototyping with built-in auth + database
- Brands that need a backend but don't warrant full AWS/GCP setup
- Projects with real-time requirements out of the box
- Developer-friendly alternative to raw cloud services

### Migration Path

```
Supabase (start)  →  Self-hosted Supabase on K8s  →  Raw AWS/GCP services (scale)
```
